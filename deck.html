真实例子 · 转换成脚本(代码可点)

11 个分区,套 find→verify 流水线

const ZONES = [ /* 11 个分区:鉴权 / 抓取 / 终判 / 前端 … 各含 files+focus */ ]

await pipeline(ZONES,
  (zone) => agent(findPrompt(zone),   {…}),          // 第一层:11 个 find
  (res)  => parallel(res.findings.map(f =>
             agent(verifyPrompt(f), {…}))))       // 第二层:每条发现 1 个 verify
🔍 点这块代码 → 看当时真实、完整的值:ZONES 全 11 项 / findPrompt / verifyPrompt / schema 逐字,以及代入后每个 agent 收到的 prompt →

我手写的只有 ZONES(11 项) 和"对每条都验一下"这条规则。上面 {…}、findPrompt(zone) 里到底是什么,点代码看真实值。

真实例子 · 模型真相

真正花钱的,不是“指挥官”

Opus 4.8 / xhigh1 个编排者
负责理解需求、拆分 workflow、决定两层 find → verify 怎么跑。
↓ 通过 agentType:'Explore' 派活
Haiku 4.581 个执行 agent
11 个 find agent 找问题;70 个 verify agent 对每条 finding 独立复核。
3,640,435tokens 全烧在 Haiku 上,不是 Opus 上
1/15约等于同量 Opus 的十五分之一成本
代价也很清楚:每条发现只由 1 个 Haiku 单票判真伪,便宜但单薄,这就是 49% 误报的结构根因。
三句话记住
  • 我写规则,不写数量 —— agent 数 = 数组长度 + 运行时数据。
  • 模板 + 数据 = 每个 prompt —— 一个模板代入每一项,长出每个 agent 的指令。
  • schema 让返回可直接用 —— 交回干净 JSON,不用再解析人话。
延伸思考 · 我对它的理解

跑通的节点,可以沉淀成代码

现在几乎每个节点都是 agent(LLM) 在跑 —— 结果会随模型本身 / 模型换代 / 随机性波动。稳定下来的步骤,可以固化成代码、以 batch 跑。

现在

🤖 find→🤖 verify

每步都问 LLM:灵活,但有波动、偏贵。

演进

⚙️ 规则/代码→🤖 只兜底疑难

稳定步骤固化成代码 batch 跑:零波动、可复现、便宜;agent 只留判断处。

其实骨架(pipeline / filter)本就是代码,agent 只在叶子;这是把「代码 vs agent」的边界继续往叶子推。这正是本项目「规则命中直接判 + LLM 只判剩下的」已在走的路。前提:该步 I/O 已稳定、可规则化 —— 别过早固化,丢掉 agent 对边界的灵活。

延伸思考 · 我对它的理解(续)

每个节点,挑「最轻的够用」的工具

不是所有节点都要 Vision(看图)。大仓库入口让 Vision 去读:容易幻觉 + 烧 token。能读 MD / grep 拿到的,就别截图看。

🖼️ Vision 看图最贵 · 易幻觉 —— 只在「必须看像素」时用
📄 文本 Agent · Read / grep / Bash便宜得多 · 更稳 —— 能读到文字就够
⚙️ 代码 / 规则最便宜 · 零波动 —— 判定完全确定时

每个节点先问:最轻的哪一级够用? 从下往上挑,别默认上 Vision。
本项目已在做:node_render 取冻结 HTML(文本)替代截图 · needsImage 分级 · 规则命中跳过 LLM。
