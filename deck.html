Workflow
用确定的代码,
编排不确定的智能

一句话需求 → 一段脚本 → 一群 sub-agent 各司其职。

组 一

三种基础积木

整个 Workflow 的 API 就三个动作,其余都是普通 JS。

先建立心智模型

我写脚本,引擎执行,每个 agent() 派一个子 LLM

我写 JS→ 引擎逐行跑→ agent() = 一个子 LLM→ 数据 return

骨架(开几个、谁先谁后)是确定的代码;每个 agent 干什么是不确定的智能。
Workflow = 用确定的代码,编排不确定的智能。

三块积木 · 全部 API 就这三个

agent · parallel · pipeline

01agent()
bot · 一个子 LLM
派 1 个子 LLM 干一件事;给 schema,就返回干净 JSON。
02parallel()
agent A agent B agent C
合并结果
一批并发,但要等最慢的也回来,才继续下一步。
03pipeline()
findverify
findverify
findverify
多阶段,每条链路各走各的;谁先 find 完,谁先 verify。

拼这三块,就能搭出后面所有 workflow 套路。

组 三

两个例子

先一个玩具,再拆本项目真发生过的一次。

简单例子 · 需求 → 脚本

"查登录/支付/搜索三个模块的安全隐患"

const MODULES = ['登录', '支付', '搜索']        // 3 项 = 3 个 agent
const SCHEMA  = { risk: string, severity: 'high|medium|low' }
await parallel(MODULES.map(m => () =>
  agent(`分析「${m}」模块最大的安全隐患`, { schema: SCHEMA })))

没写"3 个 agent"。数组有 3 项,才有 3 个。改 5 项就 5 个。

简单例子 · 代入 → 返回

一个模板 × 3 个值 = 3 份结构化结果

支付 那个 agent 收到

分析「支付」模块最大的安全隐患

它返回(按 schema)

{ risk: "金额客户端算、服务端没复核…", severity: "high" }

登录 → 账号枚举(medium);搜索 → 注入(high)。各判各的,一次并行拿回三份。

真实例子 · 转换成脚本(代码可点)

11 个分区,套 find→verify 流水线

const ZONES = [ /* 11 个分区:鉴权 / 抓取 / 终判 / 前端 … 各含 files+focus */ ]

await pipeline(ZONES,
  (zone) => agent(findPrompt(zone),   {…}),          // 第一层:11 个 find
  (res)  => parallel(res.findings.map(f =>
             agent(verifyPrompt(f), {…}))))       // 第二层:每条发现 1 个 verify
🔍 点这块代码 → 看当时真实、完整的值:ZONES 全 11 项 / findPrompt / verifyPrompt / schema 逐字,以及代入后每个 agent 收到的 prompt →

我手写的只有 ZONES(11 项) 和"对每条都验一下"这条规则。上面 {…}、findPrompt(zone) 里到底是什么,点代码看真实值。

三句话记住
  • 我写规则,不写数量 —— agent 数 = 数组长度 + 运行时数据。
  • 模板 + 数据 = 每个 prompt —— 一个模板代入每一项,长出每个 agent 的指令。
  • schema 让返回可直接用 —— 交回干净 JSON,不用再解析人话。
延伸思考 · 我对它的理解

跑通的节点,可以沉淀成代码

现在几乎每个节点都是 agent(LLM) 在跑 —— 结果会随模型本身 / 模型换代 / 随机性波动。稳定下来的步骤,可以固化成代码、以 batch 跑。

现在

🤖 find→🤖 verify

每步都问 LLM:灵活,但有波动、偏贵。

演进

⚙️ 规则/代码→🤖 只兜底疑难

稳定步骤固化成代码 batch 跑:零波动、可复现、便宜;agent 只留判断处。

其实骨架(pipeline / filter)本就是代码,agent 只在叶子;这是把「代码 vs agent」的边界继续往叶子推。这正是本项目「规则命中直接判 + LLM 只判剩下的」已在走的路。前提:该步 I/O 已稳定、可规则化 —— 别过早固化,丢掉 agent 对边界的灵活。

延伸思考 · 我对它的理解(续)

每个节点,挑「最轻的够用」的工具

不是所有节点都要 Vision(看图)。大仓库入口让 Vision 去读:容易幻觉 + 烧 token。能读 MD / grep 拿到的,就别截图看。

🖼️ Vision 看图最贵 · 易幻觉 —— 只在「必须看像素」时用
📄 文本 Agent · Read / grep / Bash便宜得多 · 更稳 —— 能读到文字就够
⚙️ 代码 / 规则最便宜 · 零波动 —— 判定完全确定时

每个节点先问:最轻的哪一级够用? 从下往上挑,别默认上 Vision。
