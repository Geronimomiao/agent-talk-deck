Tool Layer Explainer

Patch 和 Editor 都是改文件，
差别在“动作语言”

模型不会直接伸手改文件。它先生成一次工具调用，再由 agent harness 解析、执行、把结果塞回上下文。公开研究已经支持一个更稳的结论：agent 的接口、编辑格式和工具菜单会影响任务表现。

回到主 Deck 抓包实验台
不是“谁读不读原文”。
两边都要读代码。

真正差别

读完以后，模型要把修改描述成某种工具参数：是“按 diff 打补丁”，还是“找到这段原文并替换”。描述方式不同，失败模式也不同。

Patch / diff 风格

模型输出一段类似补丁的文本：这里删什么、加什么、上下文大概在哪。它更像程序员平时 review 的 diff。

描述改动 
*** Begin Patch
*** Update File: app.js
@@
-const name = "Tom";
+const name = "Jerry";
*** End Patch

Editor / 精确替换风格

模型输出原文片段和新片段：harness 去文件里找 old_string，再替换成 new_string。它更像一次字符串替换。

定位原文 
{
  "file_path": "app.js",
  "old_string": "const name = \"Tom\";",
  "new_string": "const name = \"Jerry\";"
}

一次工具调用里，模型要过四关

所以 tool calling 不只是“会调用函数”。它还包含选工具、填参数、读结果、失败恢复。
1
选对工具该用 patch、edit、bash，还是先读文件？选错工具会多走一轮。
2
写对参数patch 格式要合法；old_string 要和文件里的文本完全对上。
3
读懂返回工具失败时，harness 返回的错误文本会变成下一轮输入。
4
换策略重试好的组合会知道该缩小上下文、重新读文件、还是改用整文件重写。

Patch 常见失败

  • patch 语法写错，harness 无法解析
  • 上下文已经漂移，补丁打不上
  • 多处修改混在一起，某一处失败导致整包失败

Editor 常见失败

  • old_string 多一个空格、少一个换行，找不到
  • old_string 在文件中出现多次，不知道替换哪一处
  • 模型抄原文时漏掉缩进，导致精确匹配失败

哪些是事实，哪些是推断？

事实：不同 agent 暴露给模型的工具 schema、参数格式、错误反馈和控制流不同，这能从抓包或工具定义里直接看到。
可观察：不同编辑接口有不同失败模式，这可以用同一个小改动反复实验。
推断：“自家模型更适配自家壳”是合理解释，但除非官方说明训练细节，或者我们做出 2×2 对照实验，否则不要说成已证明的训练事实。

公开研究怎么支撑这个论证？

它们不能直接证明某家私有模型一定适配某家私有工具，但能证明“接口不是中性的”。
SWE-agent：Agent-Computer Interface 研究明确讨论了 agent-computer interface 的设计会影响软件工程 agent 的行为和表现。 SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering
To Diff or Not to Diff? 研究代码编辑输出格式，指出传统 diff 格式存在脆弱 offset / fragmented hunk 等问题，并提出结构化 diff。 Structure-Aware and Adaptive Output Formats for Code Editing
Diff-XYZ 对不同 diff 表示法做 benchmark，结论指向：不同格式适合不同 use case 和模型规模。 Diff-XYZ: A Benchmark for Evaluating Diff Understanding
ToolMenuBench 研究可见工具菜单如何影响 agent 的成功率、错误工具调用、风险暴露和 token 使用。 Benchmarking Tool-Menu Filtering Strategies

怎么验证这页不是玄学？

验证重点不是证明某家一定更强，而是证明“工具动作语言会改变失败模式”。
看工具 schema抓包里找 tools / tool_call：看模型到底要填哪些字段，字段约束是什么。
造一个小改动比如改一行变量名、改两处同名文本、改一段带缩进的 JSX，观察失败方式。
看重试链失败后模型是重新读文件、换工具、缩短 old_string，还是直接整文件重写。
这页用于解释主 deck 的“工具与编辑格式层”。它不声称某个模型一定被某个私有工具专门训练过；更稳的结论是：模型看到的是 harness 提供的动作语言，而动作语言会影响工具调用稳定性。
