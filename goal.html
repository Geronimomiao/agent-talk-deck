Codex Goal · 控制流细讲

Goal
把一句目标,
跑成持续任务循环

它不是换了一个更聪明的模型,而是 harness 多持有了一份状态:目标、预算、进度、完成判据。

每页可用 #页码 定位,例如 goal.html#6
组 一

先抓住一句话

Goal 的核心不是"自动干活",而是"有一个可检查的完成标准一直跟着线程走"。

心智模型

目标不是放在模型脑子里,是放在线程状态里

/goal 目标→ ThreadGoal 状态→ 下一轮注入上下文→ 继续推进

模型每一轮仍然只看到当前上下文。持续性来自外层系统:把 objective 存起来,后续轮次再把它作为任务上下文喂回模型。

参考: Codex manual · Goal mode · GoalStore
公开手册怎么说

Goal text 同时是起点,也是完成标准

起点

用户用 /goal 设置一段 objective;Codex 用它决定下一步做什么。

完成标准

Goal 写得越可验证,系统越容易判断"还要不要继续"。

/goal Finish the migration and keep tests green

手册也提到:如果 slash command 里没有 /goal,可以启用 [features] goals = true。

组 二

源码里的三块

工具、状态、prompt 注入。把这三块拼起来,Goal 才像"会自己接着干"。

第一块 · 工具入口

模型能碰到的,是三个 goal tool

create_goal

创建 active goal;如果已有未完成 goal,会失败。

get_goal

读取当前目标、状态、token/time 用量和剩余额度。

update_goal

只能标记 complete 或 blocked;暂停/恢复由用户或系统控制。

这里很关键:Goal 不是让模型任意改状态,而是把"能做什么状态迁移"收进工具 schema 和 handler。

源码: ext/goal/src/spec.rs · ext/goal/src/tool.rs
第二块 · 持久化状态

ThreadGoal 是一条线程级记录

ThreadGoal {
  thread_id,
  objective,
  status,
  token_budget,
  tokens_used,
  time_used_seconds,
  created_at,
  updated_at
}

active / paused

还能继续,或被用户暂停。

blocked / limited

卡住、预算到顶、用量受限。

complete

目标真的完成,不再自动继续。

源码: ThreadGoal protocol · state runtime
第三块 · 重新注入 prompt

每次继续,Goal 都会变成一段内部上下文

读 ThreadGoal→ 渲染 continuation.md→ InternalContextSource: goal→ 模型继续工作 
Continue working toward the active thread goal.

objective = 用户设置的目标
budget = tokens used / token budget / remaining
rules = 继续推进、证据验证、不要缩小目标

这就是"持续性"的来源:不是长记忆魔法,是每一轮都把目标和约束重新放进模型可见的上下文。

源码: steering.rs · continuation.md
组 三

它怎么跑起来

Goal 不是单次工具调用,而是一段线程生命周期。

执行链路

从设置目标,到下一轮自动推进

User / UI
/goal→pause / resume / edit
App Server
thread/goal/set→thread/goal/updated
State
写 SQLite→累计 token/time
Runtime
注入 continuation→开启下一轮
Model
读目标→调用工具→验证当前状态

所以你看到的"它接着干",背后其实是 UI / server / state / prompt / tools 几层一起配合。

停止条件

Goal 最贵的一点:它要求 evidence-gated completion

complete

只能在目标真的达成、且无剩余必需工作时标记。

或
blocked

同一个阻塞连续重复至少 3 个 goal turn,且没有可继续推进的动作。

这也是它和普通自动循环的区别:它不该因为"看起来差不多"就停,也不该第一次遇到困难就交还。

源码: completion / blocked audit prompt
组 四

设置场景怎么执行

拿"启用 goals"或"改默认模型"这种设置项,看它是怎么穿过 harness 的。

场景 · 设置 goals 开关

设置不是模型能力,是配置层影响运行时能力

你改设置→ ~/.codex/config.toml→ config loader→ features.goals→ 暴露 /goal 与 goal tools 
[features]
goals = true

手册里这句很有代表性:如果 UI 里没有 /goal,启用 feature flag。也就是说"能不能用 Goal"先由配置层决定,然后才轮到模型在工具边界内推进。

参考: Codex manual · config / Goal mode · features/src/lib.rs
同样的逻辑 · 改模型/沙箱/审批

配置进 harness,再影响每一轮执行

CLI flags→ project .codex/config.toml→ profile→ user config→ defaults

model

决定下一轮请求用哪个模型。

sandbox_mode

决定命令能读写哪里、网络是否受限。

approval_policy

决定什么动作要停下来问人。

所以"设置"不是一段提示词,而是 harness 在发请求、暴露工具、执行命令前读取的一组规则。

参考: Codex manual · Config basics

Goal 和 Workflow 都是控制流,但厚度不同

Workflow

你写确定的代码骨架:谁并发、谁串行、怎么合并。

Goal

系统持有一个完成标准:每轮推进、验证、继续或停止。

它们都属于 harness:模型负责每一轮推理,外层系统负责把很多轮串成一个任务。
