心智模型 五个原语 agent 数从哪来 并行 vs 流水线 并发上限 模型/推理档 脚本↔存档

Workflow 是怎么运行起来的 不是"选一个模板",而是我当场写一段 JS,引擎逐行执行它 —— 每遇到一次 agent() 就派一个子 LLM 去干活

我(主循环)

① 写一段 JS 脚本

不是 JSON、不是配置。是带 meta + agent()/parallel()/pipeline() 的普通 JavaScript。控制流(循环/条件/fan-out)由我决定。
▼ 把整段 script 交给 Workflow 工具
引擎

② 逐行执行这段脚本

脚本体在一个 async 环境里跑。await agent(...) 时,引擎 spawn 一个独立子 agent(一个新的 LLM 会话)。
▼ 每个 agent() 调用 = 一个子进程级的 LLM
子 agent × N

③ 每个子 agent 独立干活

拿到 prompt,自己用 Read/Grep/Bash 等工具查、想、跑,最后返回结果(给了 schema 就返回校验过的 JSON)。彼此互相看不见。
▼ 返回值 = 数据(不是给人看的话)
数据回到脚本

④ 脚本汇总,return 给我

纯 JS 把结果过滤/排序/合并,return 一个干净对象。我(主循环)拿到它,再决定下一步动作。全程存档成 wf_*.json。
一句话抓住本质:骨架(要开几个 agent、谁先谁后、怎么合)是确定性的代码;每个 agent 内部做什么是非确定性的智能。Workflow 就是"用确定的代码,编排不确定的智能"。

01五个原语 —— 全部编排都由它们拼出

你不需要学一门新语言。整个 Workflow 的 API 就下面 5 个函数,其余全是普通 JS。

agent(prompt, opts?)
派 1 个子 agent。给 schema 就强制返回结构化 JSON;给 agentType 换 agent 类型(连带换模型)。这是唯一"真花钱"的原语。
const r = await agent('查X', {schema:S})
parallel([fn, fn, …])
屏障式并发:同时跑一批,等全部返回才继续。要"所有结果凑齐再算"时用。
await parallel(items.map(x=>()=>agent(...)))
pipeline(items, s1, s2, …)
无屏障流水线:每个 item 独立穿过所有阶段,互不等待。多阶段的默认选择,墙钟最短。
await pipeline(items, find, verify)
phase(title) / log(msg)
纯展示:给进度树分组、给你打进度行。不影响逻辑,不花钱。
phase('Verify'); log('已确认 36 条')

02agent 数目从哪来 —— 没有"数量字段",是控制流的结果

很多人以为有个 agentCount:81 之类的开关。没有。存档里的 agentCount 是引擎跑完数出来的回执。真正的"旋钮"是你 map 的数组长度。拖下面试试:

ITEMS.length = 3
const ITEMS = ['a','b','c']   // ← 就是这里。加一项 = 多一个 agent
await parallel(ITEMS.map(x => () => agent(`处理 ${x}`)))
→ 引擎数出来:agentCount === ITEMS.length === 3  ·  没有任何地方写死这个数字

静态

数组长度 / 写几次 agent()。如上,你说了算。

动态

运行时数据决定。audit:Verify 数 = Find 实际找出的 70 条,跑之前不知道。

循环到某条件

while(找到<10) 或 while(预算>50k),一直派到满足。

真实场景:audit 那次 81 个 agent 的「两层扇出」

上面的滑块是最简单的单层(一个数组 → 数组多长就多少 agent)。但 81 那种大扇出是两层,而且我只手写了第一层: 第一层 = 我写的 ZONES(11 项,下面每张蓝卡); 第二层 = 每个 find agent 运行时跑出来的 findings,我只写了「对每条都验一下」的 .map,没写这些 item。点播放,看第二层怎么冒出来把 11 撑到 81:

当前 agent 数:11 — 只有第一层(你写的 ZONES)
确认有效 0 误报驳回 0 第二层还没跑

await pipeline(ZONES,

② findPrompt(zone) —— 11 个 find agent 各自收到的 prompt(模板 × 每个 zone 的值)

function findPrompt(zone) {
  return `你是资深代码审查者，正在审计「示例评估看板项目」项目（生产中的公开脱敏示例）。
主链路：读取任务 → 抓取页面/接口结果 → 规则+LLM 终判 → 写回报告。
项目根目录：${ROOT}

【你负责的审查分区：${zone.label}】
请用 Read 工具完整读取以下文件（必要时 Grep 它们调用/被调用的上下文，但审查范围聚焦这些文件）：
${zone.files.map(f => `  - ${ROOT}/${f}`).join('\n')}

【重点关注】${zone.focus}

【要求】
- 找真实的漏洞、缺陷、需重构处。优先正确性 bug、安全漏洞、资源泄漏、并发竞态、错误处理缺口。
- 每条发现必须引用具体代码作为 evidence（行号 + 关键片段），不能泛泛而谈、不能臆测。
- 宁缺毋滥：没把握成立的不要报。重构类(refactor)只报真正影响正确性/可维护性的，不报纯风格/命名。
- 注意这是生产工具，已经过多轮人工 + CodeX review，明显的低级错误大概率已被修过——你要找的是被遗漏的、深层的、边界的问题。
- 如果某文件确实没发现实质问题，findings 里就别硬凑。

返回结构化 findings 数组。`
}

③ verifyPrompt(f) —— 70 个 verify agent 各自收到的 prompt(f = 上一层某条 finding)

function verifyPrompt(f) {
  return `你是严格的代码审查验证者，任务是【对抗性地验证】下面这条审查发现是否真实成立。默认怀疑它是误报，除非代码证据确凿。
项目根目录：${ROOT}

【待验证的发现】
标题：${f.title}
文件：${f.file}
行号：${f.line}
严重度：${f.severity}（${f.category}）
描述：${f.description}
证据：${f.evidence}
建议：${f.suggestion}

【你必须做的】
1. 用 Read 工具打开 ${ROOT}/${f.file}，定位到 ${f.line} 附近的真实代码。
2. 核对：审查者引用的代码是否真实存在？逻辑是否真如其所述？有没有它没看到的上下文（如上游已校验、调用方已处理、有 try-catch 兜底、测试已覆盖）使这个"问题"实际不成立？
3. 必要时 Grep 调用链、相关测试文件（test/ 目录）确认是否已有防护或已被处理。
4. 给出裁决：isReal=true 仅当你独立确认这确实是个真问题；isReal=false 若它是误报、被上下文化解、或基于对"净 diff/既有代码"的误读。
reasoning 必须引用你实际看到的代码，不能复述原描述。`
}

④ 那两个 {…} 到底是什么 —— 真实的 pipeline 调用 + agent opts

const zoneResults = await pipeline(
  ZONES,
  // stage 1: 分区审查
  (zone) => agent(findPrompt(zone), {
    label: `审:${zone.key}`,
    phase: 'Find',
    schema: FINDINGS_SCHEMA,
    agentType: 'Explore',
  }).then(r => ({ zone, findings: (r && r.findings) || [] })),
  // stage 2: 对该分区每条发现并行对抗验证
  (res) => {
    if (!res || !res.findings.length) return { zone: res ? res.zone : null, verified: [] }
    return parallel(res.findings.map(f => () =>
      agent(verifyPrompt(f), {
        label: `验:${f.file.split('/').pop()}:${f.line}`,
        phase: 'Verify',
        schema: VERDICT_SCHEMA,
        agentType: 'Explore',
      }).then(v => ({ ...f, verdict: v })).catch(() => ({ ...f, verdict: null }))
    )).then(verified => ({ zone: res.zone, verified }))
  }
)

⑤ opts 里的 schema 引用的两个校验模板(决定 agent 必须返回什么形状)

const FINDINGS_SCHEMA = {
  type: 'object',
  properties: {
    findings: {
      type: 'array',
      items: {
        type: 'object',
        properties: {
          title: { type: 'string', description: '一句话问题标题' },
          file: { type: 'string', description: '文件路径' },
          line: { type: 'string', description: '行号或行号范围（尽量精确）' },
          severity: { type: 'string', enum: ['high', 'medium', 'low'], description: 'high=正确性bug/安全漏洞/数据损坏；medium=边界/健壮性缺口；low=重构/可维护性' },
          category: { type: 'string', enum: ['bug', 'security', 'resource-leak', 'race', 'error-handling', 'refactor'] },
          description: { type: 'string', description: '问题是什么、为什么是问题、触发条件' },
          evidence: { type: 'string', description: '引用的关键代码片段（证明问题存在）' },
          suggestion: { type: 'string', description: '建议怎么修' },
        },
        required: ['title', 'file', 'line', 'severity', 'category', 'description', 'evidence', 'suggestion'],
      },
    },
  },
  required: ['findings'],
}

const VERDICT_SCHEMA = {
  type: 'object',
  properties: {
    isReal: { type: 'boolean', description: '这个问题是否真实成立（true=确认有效，false=误报/不成立）' },
    confidence: { type: 'string', enum: ['high', 'medium', 'low'] },
    reasoning: { type: 'string', description: '为什么成立或为什么是误报（必须引用实际代码，不能凭空）' },
    severityAdjust: { type: 'string', enum: ['keep', 'upgrade', 'downgrade'], description: '严重度是否需要调整' },
  },
  required: ['isReal', 'confidence', 'reasoning', 'severityAdjust'],
}

03parallel vs pipeline —— 最关键、最容易忽略的区别

两者都"并发",区别在阶段之间要不要设屏障。下面 3 个 item(A/B/C)耗时不同,点▶ 播放看两种模式的墙钟差多少。红色斜纹 = 被屏障白白等掉的时间。

pipeline(无屏障)
parallel(屏障)
■ find 阶段   ■ verify 阶段
本模式墙钟:—
另一模式:—

pipeline:A 验的时候 B 还在审

每个 item 一审完立刻进验证,不等别人。总耗时 = 最慢的那条"审+验"链。这是多阶段的默认选择。

parallel:所有人卡在屏障上

必须等最慢的 find 全好,才统一开 verify。快的 item 干等着。总耗时 = 最慢find + 最慢verify。只在"要凑齐所有结果再算"时才值得。

04并发上限 —— 派 100 个不等于同时跑 100 个

脚本里写 100 次 agent() 没问题,但引擎同时最多跑 ≈ min(16, 核数−2) 个,其余排队,有空位就顶上。所以大扇出是一条流水,不是一次齐发。

40 个 agent
■ 跑 ≤16   ■ 排队 其余  ·  上限满了,新 agent 等前面的腾出槽位

05子 agent 用什么模型 / 推理档

默认继承主循环(会话)的模型和 effort;脚本可按 agent 覆盖。以下是从脱敏示例存档实测出来的对应关系:

脚本里怎么写实际模型推理档 effort
不传 agentType / model继承会话模型(如 opus-4-8)继承会话 effort(如 xhigh)
agentType:'Explore'haiku-4-5(Explore 自带,覆盖继承)—
agentType:'general-purpose'/'Plan'继承会话模型(opus-4-8)继承
显式 model:'…' / effort:'high'按你写的来按你写的来
设计要点:广度活(通读找线索)派 Explore → 走便宜的 Haiku,可放心开几十上百个;深度活(对抗验证、方案批判)继承 Opus、按需 effort:'high'。这两个都写在脚本的 opts 里,不是配置文件。

06脚本 ↔ 存档:你只写左边,右边是引擎记的账

跑完引擎存一个 wf_*.json。它 = 你的 script + 运行回执。你手写的只有 script 一个字段(还是 JS,不是 JSON)。

你写的(script 字段)

export const meta = { name, description }
const ITEMS = [...]              // 决定 agent 数
const r = await parallel(
  ITEMS.map(x => () => agent(...)))
return r                         // → 进 result 字段

引擎记的账(其余 16 字段)

script ← 你的 JS
result ← 你 return 的值
agentCount · totalTokens · durationMs
runId · status · defaultModel
workflowProgress · phases · …
这些全是跑完自动填的回执,改它们没有意义。
