01扇出结构:81 是怎么来的

核心算式很简单 —— 11 个分区各 1 个 Find agent,它们共找出 70 条候选; 每条候选再派 1 个 Verify agent 独立复核。11 + 70 = 81。

workflow 脚本 · 主循环写好后交给引擎执行
pipeline(ZONES, findStage, verifyStage)
▾
阶段 1 · Find · 11 个 agent 并行
每个 agent 拿到 1 个分区:分配的文件清单 + 聚焦点(如鉴权区专盯「恒定时间比较 / fail-open / 身份伪造」),用 Read Grep Bash 通读后交结构化 findings。
▾ pipeline:某分区一审完,立刻验它自己的发现,不等别的分区
阶段 2 · Verify · 70 个 agent(每条候选 1 个)
■ 确认有效 36  ·  ■ 驳回为误报 34。 每个 Verify agent 的 prompt 都写死「默认怀疑它是误报,除非代码证据确凿」,必须重新打开文件独立复核。
▾
阶段 3 · Synthesize · 纯 JS,不开 agent
把 isReal=false 的 34 条滤掉、按严重度排序、应用严重度调整 → return 36 条给主循环的我。

02脚本原理:三个原语撑起全部扇出

整个编排的核心就下面这十几行。控制流是确定性的普通 JS(循环 / 条件 / fan-out), 真正的「智能」发生在每个 agent() 里 —— 它 spawn 一个独立子 LLM 去干活。

// ZONES = 11 个分区,每个 { key, label, files:[...], focus }
const zoneResults = await pipeline(ZONES,

  // ── 阶段1 findStage:每个分区 → spawn 1 个 Find agent ──
  (zone) => agent(findPrompt(zone), {
    label: `审:${zone.key}`,
    phase: 'Find',
    schema: FINDINGS_SCHEMA,   // ← 强制返回结构化 JSON,校验不过自动重试
    agentType: 'Explore',     // ← 只读 agent,不能改代码
  }),

  // ── 阶段2 verifyStage:该分区每条 finding → spawn 1 个 Verify agent ──
  (res) => parallel(res.findings.map((f) => () =>
    agent(verifyPrompt(f), {
      phase: 'Verify',
      schema: VERDICT_SCHEMA,  // ← { isReal, confidence, reasoning }
    }).then((v) => ({ ...f, verdict: v }))
  ))
);

// ── 阶段3:纯 JS 汇总,零 agent ──
const confirmed = zoneResults.flat().filter(f => f.verdict.isReal);
agent()
起一个子 agent。给了 schema 就强制它最后调 StructuredOutput 交合规 JSON —— 这就是为什么返回值能直接当数据用,不用解析自然语言。
pipeline()
无屏障流水线:每个 zone 独立穿过「find→verify」两阶段,A 区在验证时 B 区可能还在审。所以 81 个 agent 的墙钟只花了 ~11 分钟,而不是串行相加。
parallel()
屏障:一个分区内的 N 条 finding 同时开 N 个 verify agent,全部返回才算这个分区完事。
schema
Find 用 findings 数组 schema,Verify 用 {isReal, reasoning} schema。类型不对在工具层就打回重试,主循环拿到的永远是干净结构。

03调度真相:81 个不是同时在跑

脚本里写了 81 次 agent(),但引擎有并发上限 ≈ min(16, 核数−2),大约同时 10~16 个在跑,其余排队,有空位就顶上。所以这是一条持续 11 分钟、总量 81 个 agent 的流水,不是一次性 81 路齐发。

为什么用 pipeline 而不是「先全审完再全验」

若用屏障(先等 11 个 Find 全好),最慢那个分区会拖住所有验证。pipeline 让先审完的分区立刻进入验证 —— 快的分区不等慢的,整体墙钟塌缩到「最慢单条 find→verify 链」而非「最慢 find + 最慢 verify」。

agent 之间互相看不见

11 个 Find agent 各审各的分区,彼此不知道对方发现了什么 —— 这正是「多视角覆盖」的来源。汇总只发生在最后的纯 JS 阶段。Verify agent 也只拿到「一条待验证发现」,不受原审查者措辞诱导。
